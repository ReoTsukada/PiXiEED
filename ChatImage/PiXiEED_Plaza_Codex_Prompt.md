# PiXiEED リアルタイム鑑賞型チャット広場 実装プロンプト

あなたは、Webゲーム、リアルタイム通信、Supabase、2Dレンダリング、負荷対策、UI/UX設計に精通したシニアフルスタックエンジニアです。

既存のPiXiEEDプロジェクトに、新しいリアルタイム鑑賞型チャットサービスを追加してください。

仮称：
PiXiEE Plaza
または
PiXiEE Gallery Chat

目的：
ユーザーが自分のちびキャラクターを操作して、真っ白な2Dアイソメトリック空間を歩き回りながら、投稿されたイラストを他の参加者とリアルタイムで鑑賞し、会話できるサービスを作る。

これは通常のチャットルームではない。

「その瞬間に表示されている作品を、同じ時間に居合わせた人たちと鑑賞する広場」であり、後から訪れたユーザーも、その作品に残された過去のコメントを見て楽しめる仕組みにする。

投稿作品は投稿から7日後に完全削除される。
いいね、フォロー、ランキング、恒久的なプロフィール評価は設けない。
一期一会と、その場の感想を中心にする。

---

## 最重要ルール

1. 既存コードを変更する前に、プロジェクト全体を調査すること。
2. 使用中の技術スタック、認証、Supabaseクライアント、共通CSS、ルーティング、PWA、既存投稿機能を確認すること。
3. 調査結果と実装計画を最初に提示すること。
4. 大規模な一括書き換えをしないこと。
5. 機能単位で小さく実装し、各段階で動作確認すること。
6. 既存のPiXiEEDraw、PiXiEELENS、認証、投稿機能を壊さないこと。
7. 秘密鍵やservice role keyをブラウザへ置かないこと。
8. セキュリティ制御は必ずSupabase RLSとサーバー側処理でも行うこと。
9. 最大100人が同一ルームに入っても、全員分の書き込みがDBへ大量発生しない構成にすること。
10. 最初から完成版を一括実装せず、MVPから段階的に進めること。

## サービスの基本構造

画面は、真っ白な2Dアイソメトリック視点の広場とする。

空間内には以下を配置する。

- 中央の巨大な作品掲示板
- 空中をゆっくり移動する白い飛行船
- 飛行船側面の作品表示パネル
- 小型の案内掲示板
- 投稿受付端末
- 課金コメント送信用のギフト端末
- 現在の参加人数表示
- 現在展示中の作品情報
- 次の作品へ切り替わるまでのカウントダウン

世界観は白を基調とし、余計な背景装飾を減らす。
投稿作品とユーザーアバターが主役になるデザインにする。

床、壁、階段、掲示板、飛行船などは白から薄いグレーで構成する。
影は淡いグレー。
UIのアクセントカラーはPiXiEEDの既存ブランドカラーを調査して利用する。

## アバター

各ユーザーは2Dのちびキャラクターとして広場に表示される。

必要な状態：

- 待機
- 上下左右または8方向移動
- 歩行アニメーション
- 発言中
- 作品を見ている状態
- ギフトコメント送信時の簡単な演出
- 切断、復帰

操作方法：

PC：
- WASD
- 矢印キー
- クリックした位置まで自動移動

スマートフォン：
- 画面上の仮想スティック
- タップ移動

他ユーザーの頭上には以下を表示する。

- 表示名
- 短時間表示される吹き出し
- 必要に応じて簡略化したステータス

ユーザーが多い場合は、遠距離にいるユーザーの名前や吹き出しを簡略化する。

## リアルタイム同期

最大100人程度の同時参加を想定する。

Supabase Realtimeを利用する場合、以下のように分離する。

永続化するデータ：

- 作品
- 作品コメント
- 有料コメント
- 通報
- ユーザー設定
- アバター設定
- 課金履歴
- 削除予定時刻

永続化しないデータ：

- アバターの現在座標
- 移動方向
- 歩行中かどうか
- 一時的な吹き出し
- 現在オンラインかどうか
- カーソルや視線相当の一時状態

永続化しない情報はSupabase Realtime PresenceまたはBroadcastを利用する。
アバター座標を毎フレームDBへ保存してはいけない。

移動同期仕様：

- クライアント描画は60fpsを目標
- ネットワーク送信は毎秒5〜10回程度
- 受信した座標は補間して滑らかに表示
- 瞬間移動を防ぐ
- 一定距離以上ずれた場合のみ補正
- 非表示タブでは送信頻度を落とす
- 復帰時に最新状態を再取得
- 切断ユーザーを自動削除
- Presence情報の上限サイズを小さく保つ

100人全員を常時高精度同期する必要はない。
画面内または近距離のユーザーを優先表示し、遠距離ユーザーは更新頻度を下げてもよい。

## 作品の表示

ルームでは、現在のメイン作品を1枚表示する。

通常作品は10秒ごとに切り替える。

ただし、10秒固定だとコメントを読む前に切り替わる可能性があるため、以下を設定可能にする。

- 基本表示時間：10秒
- コメントが活発な場合：最大30秒まで延長可能
- ユーザーが詳細表示を開いた場合：そのユーザーの画面だけ静止可能
- ルーム全体の作品切り替え時刻はサーバー時刻を基準に統一

作品表示場所：

1. 中央の大型掲示板
2. 飛行船側面の大型スクリーン
3. 作品詳細モーダル
4. 画面端の小型プレビュー

飛行船はゆっくり広場上空を移動し、現在展示中の作品または次に表示予定の作品を表示する。

## 作品キュー

作品の表示順には優先度を設ける。

最優先：

- その場で新しく投稿された作品

次点：

- まだ十分に表示されていない作品
- コメント数が少ない作品
- 最近投稿された作品

禁止：

- 人気順固定
- いいね順
- 課金額による常時優遇
- 同じ投稿者の連続表示
- 特定ユーザーだけが長時間独占する仕様

作品キューには公平性を持たせる。

推奨カラム：

plaza_artworks

- id
- user_id
- storage_path
- thumbnail_path
- title
- width
- height
- mime_type
- file_size
- created_at
- expires_at
- deleted_at
- display_count
- last_displayed_at
- priority_until
- moderation_status
- room_id

新規投稿時：

priority_untilに短い優先期間を設定する。
ただし連投による占有を防ぐため、投稿者単位のクールダウンを設ける。

## コメント

発言には2種類ある。

### 1. リアルタイム吹き出し

その場にいるユーザーのアバター上へ短時間表示する。

### 2. 作品コメント

現在表示中の作品へ保存される。
同じ作品を後から見たユーザーも閲覧できる。

通常のチャット送信時は、原則として現在表示中の作品IDへ紐付ける。

コメントテーブル案：

plaza_artwork_comments

- id
- artwork_id
- user_id
- body
- comment_type
- created_at
- expires_at
- is_paid
- gift_type
- moderation_status

作品が削除される際、コメントも一緒に削除する。

コメント表示には以下を用意する。

- 現在その場で発言されたコメント
- 過去に同じ作品へ付いたコメント
- 発言時刻
- 投稿者名
- 通常コメントかギフトコメントか
- 通報ボタン

過去コメントとリアルタイムコメントは見分けられるようにするが、UIを分断しすぎない。

## 有料コメント

いいね機能は実装しない。

作品が良いと思ったユーザーは、課金アイテムを消費して「作者へ通知されるコメント」を送信できる。

名称案：

- 光るコメント
- お届けコメント
- ギフトコメント
- 星便
- アートレター

有料コメントの仕様：

- 通常コメントより少し目立つ
- 作者へ通知される
- 作者がオフラインでも通知一覧で確認可能
- 金額や課金アイテム数を必要以上に強調しない
- ランキングへ反映しない
- 作品の表示順位を恒久的に上げない
- 送信前に確認画面を表示
- 二重送信を防止
- 課金処理はサーバー側で検証
- 残高減算とコメント作成を一つの安全な処理として実行
- 失敗時には残高だけ減らないようにする

最初のMVPでは実決済を入れず、テスト用ポイントで動作確認できるようにする。
決済は別フェーズで実装する。

## 投稿

広場内の投稿受付端末またはUIボタンから画像を投稿できる。

対応形式：

- PNG
- JPEG
- WebP
- GIFは初期段階では静止画化または非対応
- PiXiEEDraw作品の直接投稿導線を将来的に追加

投稿時に行うこと：

- ログイン確認
- ファイル形式検証
- MIME検証
- 容量制限
- 画像サイズ制限
- 画像デコード確認
- 危険なメタデータの除去
- サムネイル生成
- 必要に応じて再エンコード
- 短時間の連投制限
- 投稿完了後、作品キューの最優先へ追加
- 広場全員へ新規投稿通知
- 次回切り替え時に優先表示

投稿直後に現在表示中の作品を強制的に途中終了させない。
原則として現在作品の表示終了後に、新規投稿作品を表示する。

## 7日後の完全削除

作品は投稿から7日後に表示対象外となり、データから削除する。

削除対象：

- 作品DBレコード
- 原寸画像
- サムネイル
- 作品コメント
- 通報関連の不要データ
- 作品に紐付く一時キャッシュ

ただし、決済記録や会計上必要な情報は作品本文と分離し、法令や運用上必要な期間だけ保持する。

実装方法：

- created_atとは別にexpires_atを保存
- expires_at = created_at + 7日
- 表示クエリでは必ずexpires_at > now()を条件にする
- 期限切れ直後は即座に非表示
- Storageの物理削除はサーバー側の定期処理で実行
- 削除失敗時は再試行
- 削除処理のログを残す
- クライアントから削除対象を自由指定できないようにする

Supabase Edge Functions、Cron、pg_cronなど、現在のプロジェクト構成に最も合う方法を調査して採用する。

RLSだけではStorageファイルの完全削除処理を行わないこと。
service role keyはEdge Functionなどの安全なサーバー環境だけで使用する。

最初は1つの共通広場だけでよい。

room_idは将来拡張のために設ける。

将来のルーム例：

- 新着作品広場
- ドット絵広場
- 初心者広場
- 共同制作広場
- イベント広場

MVPでは複数ルームUIを作り込みすぎない。

最大100人を超えた場合は、同じ展示キューを共有する複数インスタンスへ分割できる設計にする。

例：

- plaza-1
- plaza-2
- plaza-3

作品キューとコメントは共有可能だが、Presenceとアバター位置はルームインスタンスごとに分離する。

## 技術方針

既存構成を確認したうえで、以下から適切なものを選ぶ。

描画：

- Phaser 3
- PixiJS
- 既存Canvas実装の拡張

推奨：

ゲーム的な移動、カメラ、スプライト、当たり判定が必要なため、既存構成に問題がなければPhaser 3を第一候補とする。
軽量な描画中心ならPixiJSでもよい。

UI：

- ゲームCanvasとDOM UIを分離
- チャット入力、モーダル、投稿フォームはDOM
- アバター、床、飛行船、掲示板はCanvas

Realtime：

- Supabase Presence
- Supabase Broadcast
- Postgres Changesは永続コメントや作品更新だけに限定

サーバー処理：

- Edge Functions
- Postgres RPC
- Cronまたはpg_cron

画像：

- Supabase Storage
- サムネイル生成
- 画像の安全な再エンコード

## 演出

新しい作品へ切り替わる時：

- 掲示板が白くフェード
- 新しい作品が静かに表示
- 飛行船のスクリーンも同期
- 作品タイトルを数秒だけ表示
- 音はデフォルトOFF
- 派手すぎる演出は避ける

ギフトコメント時：

- 送信者から掲示板へ小さな光が飛ぶ
- 作者への通知アイコンが出る
- 他ユーザーの画面を邪魔しない
- 課金額を広場全体に大きく表示しない

新規投稿時：

- 投稿受付端末が一度だけ発光
- 「新しい作品が届きました」と小さく表示
- 次の表示候補へ追加

2. 関係するファイルを列挙する

## Codex検証メモ 2026-06-25

- `localhost:8001` への通常権限curl失敗だけで、ローカルサーバー停止とは判断しない。サンドボックス制限で接続確認だけ失敗する場合がある。
- `PORT=8001` 起動が `EADDRINUSE` の場合は、固定ポートにこだわらず空きポートを使う。
- 広場検証は `node scripts/verify-plaza.mjs` を使う。このスクリプトは空きポートで配信し、PC/スマホ幅、非スクロール、AdSenseフッター、ズーム、タップ移動をまとめて確認する。
- 自キャラDOMは `#localAvatar` ではなく `.plaza-avatar.is-self`。存在しないID待ちでタイムアウトさせない。
- PC幅では右チャットパネルがシーン右側を覆うため、クリック検証はシーン左寄り座標を使う。

## Codex実装メモ 2026-06-25 投稿削除

- 作品詳細に投稿者本人だけが見える削除ボタンを追加する。
- 削除はクライアントからStorage物理削除を直接行わず、`plaza_artworks.deleted_at` と `moderation_status='deleted'` を更新して即時非表示にする。
- Storageの完全削除は、当初設計通りサーバー側の定期削除処理で扱う。
- Realtimeの作品監視はINSERTだけでなくUPDATE/DELETEも扱い、削除・非表示更新を受けた作品を広場から外す。

## Codex実装メモ 2026-06-25 7日削除

- `supabase/functions/plaza-cleanup-expired` を追加する。
- FunctionはDB上の期限切れ作品、または `deleted_at` / `moderation_status='deleted'` の作品だけを対象にして、対応する `plaza/` Storageパスを削除する。クライアントから任意Storageパスは指定させない。
- Function呼び出しには `PIXIEED_PLAZA_CLEANUP_SECRET` を必須にする。`x-pixieed-plaza-cleanup-secret` ヘッダーまたは `secret` queryで渡す。
- 先に `dryRun=true` で対象件数とStorageパスを確認できる。
- Deploy例:
  - `supabase secrets set PIXIEED_PLAZA_CLEANUP_SECRET=... --project-ref kyyiuakrqomzlikfaire`
  - `supabase functions deploy plaza-cleanup-expired --project-ref kyyiuakrqomzlikfaire --no-verify-jwt --use-api`

## Codex実装メモ 2026-06-26 歩行と吹き出し

- ギフトコメントは現時点のMVPから外す。マップ上の `GIFT` 端末は `CHAT` 端末へ変更し、チャットパネルを開く導線にする。

- コメント送信時はコメント欄への追加だけでなく、自分のアバター上に吹き出しとして表示する。Realtime接続中は他参加者にもBroadcastする。

## Codex実装メモ 2026-06-26 カメラ追従

- カメラは常に自分のキャラクターをビューポート中央に捉える。マップ端でもカメラをワールド範囲でクランプしない。
- マップクリック/タップは引き続き `screenToWorld` でワールド座標へ変換し、目的地をワールド内に丸めてA*移動する。
- ズーム時もクリック位置中心ではなく、自キャラ中央のまま倍率だけ変える。
- `scripts/verify-plaza.mjs` は自キャラのワールド座標がシーン中央に描画されていることも検証する。

## Codex実装メモ 2026-06-26 アバターとオフライン

- 自分のキャラクターは現時点ではプロフィールで選んだアバター画像をそのまま使う。検証では `.plaza-avatar.is-self .plaza-avatar__sprite` が `character-dots/` の画像を参照することを確認する。
- オフラインでも広場画面、ランダムスポーン、マップクリック/タップ移動、ズーム、自分の吹き出し表示はローカルで使える。
- オフライン時はRealtime同期、投稿保存、コメント永続保存は行わない。画面ステータスは「オフライン表示中。移動と吹き出しは使えます」とする。

## Codex不具合メモ 2026-06-26 file直開き

- `contest/index.html` を `file://` で直接開いた場合、`type="module"` の `plaza.js` はChromeのCORS制限で読み込まれない。
- JSが読み込まれないと、自キャラ生成、カメラ更新、クリック移動、ボタン操作がすべて止まり、マップ左上固定のように見える。
- 対策として `plaza.js` は通常の `defer` scriptとして読み込む。Supabaseは内部でdynamic importするため、オンライン時のみ初期化され、失敗してもローカル移動と吹き出しは動く。
- AdSenseの403/400はfile/local確認時には起きうるが、広場JS停止とは別件。
- `scripts/verify-plaza.mjs` はfile直開きでも自キャラ、移動、吹き出しが動くことを確認する。

## Codex実装メモ 2026-06-26 横スクロールマップ

- 広場はアイソメトリック風の自由移動マップではなく、本当の2D横スクロールアクション風マップにする。
- MVPのワールドサイズは `5200x760px`。自キャラの足元座標は地面上端の `PLAYER_LANE_Y=600` に固定する。
- 画面クリック/タップは目的地のX座標だけを使う。Y座標は常に地面の歩行ラインへ丸める。
- キーボード移動は左/右のみ。`ArrowLeft` / `A` と `ArrowRight` / `D` を使い、上下キーは無視する。
- 以前のA*パスファインダー前提は、この横スクロールMVPでは使わない。縦移動マップへ戻す場合だけ再検討する。
- カメラは横方向では自分のキャラクターを中央に捉える。縦方向はキャラを中央に置かず、地面が画面下寄りに見える横スクロール構図にする。
- 背景には大きな画像表示スペース、遠景帯、遊覧飛行のライン、ゆっくり飛ぶ遊覧飛行船を置く。将来、ここをユーザー作成のドット絵背景やObjectへ差し替える。
- スポーン地点は横レーン上の複数範囲からランダムに選ぶ。アイソメトリック用の障害物判定ではスポーンを除外しない。
- `scripts/verify-plaza.mjs` は、PC/スマホ/file直開きに加えて、タップ移動後も自キャラのワールドY座標が変わらないこと、足元が地面ライン上にあること、地面が画面下寄りに表示されることを検証する。

## Codex設計メモ 2026-06-26 広告と作品表示

- 作品をサムネイル一覧や画像表示欄として並べる場合、画像5枚ごとに1回だけ広告スロットを挟む構成がよい。
- ただし、ゲームのメインビューポート内へ頻繁に広告を挿入すると移動と鑑賞を邪魔するため、広告はサイドの画像リスト、詳細モーダル下部、または作品履歴パネル側に置く。
- 横スクロールマップ上の大型掲示板や飛行船スクリーンは作品そのものを主役にし、広告はそこへ混ぜない。
