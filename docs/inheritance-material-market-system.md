# PiXiEEDマーケット 継承型素材販売システム 最新実装仕様

- 更新日: 2026-07-22
- 対象: 本番マーケット、出品、購入、派生販売、決済、分配、表示報酬、プレイ記録、出金、運営
- 基準: 2026-07-19時点のリポジトリ実装と本番Supabase構成
- 関連構想: [継承型素材販売システム 最終整理案](./inheritance-material-market-system-final-proposal.md)

本書は構想だけを説明する資料ではなく、現在の本番実装、利用者に見える動作、DB・RPC・Edge Functionの境界、未接続機能、既知の不整合を一つにまとめた運用仕様書である。構想資料と食い違う場合、現在利用できる機能の判断には本書を優先する。

## 1. 用語

| 用語 | 本書での意味 |
|---|---|
| 販売価格 | 出品者が商品本体に設定する金額 |
| 有料オプション | 商用利用、グッズ利用、クレジット省略、限定販売、カスタム条件などに加算される金額 |
| 購入価格 | 販売価格と選択済み有料オプションの合計。Stripeで購入者が支払う総額 |
| 販売収益 | 現在の商品販売者へ配分される売上分。総支払額そのものではない |
| PiXiEED利用手数料 | PiXiEEDが販売価格帯に応じて控除するプラットフォーム手数料 |
| 決済実費 | Stripeの決済、Transfer、入金、返金、チャージバック等に伴う実費 |
| 継承ロイヤリティー | 派生商品の販売時に、その商品の祖先作者へ系列に沿って配分される対価 |
| 表示報酬 | 商品の有効表示数に応じて作者へ配分する報酬。原資はPiXiEEDの売上の一部 |
| プレイ報酬 | PiXFiND等の有効プレイ数に応じて作者へ配分する予定の報酬。原資はPiXiEEDの売上の一部 |
| 報酬予算 | PiXiEEDの売上の一部から、対象月の表示報酬またはプレイ報酬へ充てる上限額 |
| 派生出品権 | 派生販売可能商品を正規取得した購入記録に付く、原則1商品分の無料出品権 |
| 系列 | 大元作品と、そこから連なる全派生作品の系譜 |
| 大元作品 | 親作品を持たない系列の起点 |
| 派生作品 | 正規購入と派生出品権を使い、親作品を改変・追加して出品した商品 |
| 管理者取得 | 運用確認のため、決済なしで管理者へ付与する`granted`取得権。販売実績には含めない |

「報酬」「ロイヤリティー」「利用手数料」は同義ではない。表示・プレイに対する作者還元は報酬、派生販売から祖先へ渡す対価は継承ロイヤリティー、PiXiEED側の控除は利用手数料と表記する。表示回数・プレイ回数に応じた報酬の原資は、PiXiEEDの売上の一部と表記する。

## 2. 現在の公開状態

- DEVアカウント限定マーケットは終了し、本番マーケットを公開している。
- DEV確認用のダミー商品・固定フィクスチャは公開一覧から撤去済み。
- 商品一覧、商品詳細、透かし付きプレビューは未ログインでも閲覧できる。
- 売り切れまたは取り下げ済みの商品は公開一覧・一般向け商品URLから外す。購入済みの利用権、系列、報酬、出品者の履歴は維持する。
- メール確認済みの通常PiXiEEDアカウントであれば、購入と出品を利用できる。
- 匿名アカウント、メール未確認アカウントは購入・出品できない。
- Stripe Connect設定は出品開始の条件ではなく、売上を出金するための条件である。
- 審査担当・管理者権限は一般ユーザーから独立し、`market_staff_roles`で管理する。

## 3. システムの目的

素材を一度売って終わりにせず、正規購入者が改変・追加した作品を、元作品との関係と利用条件を維持したまま再販売できる市場を作る。

- 大元作品は、派生販売を許可するか選べる。
- 派生販売の許可料金は0円固定であり、購入価格へ追加しない。
- 派生販売可能商品の購入記録1件に、派生商品を原則1件出品できる権利を作る。
- 派生作品の販売時は、現在の販売者だけでなく祖先作者へ継承ロイヤリティーを配分する。
- 商品の有効表示に対する表示報酬は、表示作品だけでなく直系系列にも配分する。
- PiXFiND等の有効プレイ数を、将来のプレイ報酬計算に利用できる形で記録する。
- 購入済み商品の利用権、系列、分配履歴を、出品取り下げ後も失わせない。

## 4. アカウントと権限

### 4.1 公開閲覧

公開商品は`market_public_catalog_v1`と`market_public_asset_v1`を通して取得する。元ファイル、非公開Storageパス、購入者情報、お気に入りしたユーザー一覧は公開しない。

### 4.2 購入者・出品者

`market_current_user_can_sell()`は、マーケットが有効で、現在のユーザーがメール確認済みの非匿名アカウントかを確認する。販売者情報、TOTP、Stripe Connectは安全な売上受取に利用するが、現行の公開マーケットではそれらが未完了でも出品下書きと公開条件判定へ進める。

### 4.3 審査担当・管理者

- 一般ユーザーは審査RPCを実行できない。
- 審査担当は検査済みファイルと申告内容を確認し、承認、却下、停止を行う。
- 管理者は月次表示報酬予算、月次試算・確定、管理者取得、販売者確認、広告OFF専用権限等を扱う。
- 管理操作はサーバー側で権限確認し、監査ログへ記録する。

## 5. 商品と系列のデータ構造

### 5.1 系列

`market_asset_series`は系列単位の不変条件を持つ。

- 大元作品IDと大元作者ID
- 系列状態
- 派生販売の許可状態
- 大元作品の基本利用価格
- 必須オプション合計
- 派生許可料金0円
- 継承される利用条件、禁止用途、利用可能媒体

派生作者は大元作品の条件を緩められない。派生販売可能な系列では、派生作品も次の購入者へ同じ派生許可を継承する。

### 5.2 商品

`market_assets`は商品ごとの状態を持つ。

- 系列ID、親商品ID、作者ID
- タイトル、説明、変更・追加箇所
- 公開用作者名スナップショット
- 販売価格、通貨、状態
- 形式、マニフェスト、ファイル・プレビュー参照
- 元ファイルSHA-256、パッケージSHA-256、制作元区分
- AI使用申告、権利確認、規約・プライバシー確認版
- タグ、お気に入り数、直接派生数
- 限定販売数、販売済み数
- 取り下げ日時、取り下げ理由

現在の系列モデルは親を1作品に限定している。複数の親素材を同時に使う多系列商品は未実装である。

## 6. 出品フロー

### 6.1 入口

- 大元作品は通常の出品画面から作成する。
- 派生作品は、マイページの購入済み商品にある派生出品導線から開始する。
- 派生画面では購入記録、派生出品権、親商品、系列がサーバー側で固定される。
- 既存のPiXiEED由来パッケージを、新しい大元作品として再登録することはできない。

### 6.2 入力内容

出品者は次を設定する。

1. ファイルまたはフォルダ
2. 販売対象形式
3. サムネイル1枚と購入前サンプル最大6枚
4. タイトル、説明、派生の場合は変更・追加箇所
5. 販売価格
6. 利用オプション、限定販売、派生許可、カスタムオプション
7. タグ最大5個
8. AI使用あり／なし
9. 自分が販売権を持つことの確認
10. 利用規約とプライバシーポリシーの版ごとの確認

現在送信する規約・プライバシー版は`2026-07-19`である。

### 6.3 下書き保存

出品画面の入力はIndexedDBへ保存する。

- DB名: `pixieed-market-listing-drafts`
- ストア: `drafts`
- 分離単位: ログインユーザー、大元／派生コンテキスト、派生権
- 保存対象: テキスト、選択、チェック、タグ、価格、通常オプション、カスタムオプション、ファイル、形式、プレビュー

リロードやブラウザ再表示後も同じアカウント・同じ出品コンテキストで復元する。送信成功後は該当下書きを消去する。

### 6.4 パッケージ確定

- 大元作品作成境界: `market_create_root_asset_v7`
- 派生作品作成境界: `market_create_derivative_draft_v4`
- 派生コンテキスト取得: `market_derivative_listing_context_v1`
- パッケージ添付: `market_attach_listing_package`

サーバーは販売価格、オプション、形式、タグ、権利、系列、AI申告、文書確認版を再検証する。同じ確定済みパッケージを再送した場合は冪等に扱い、二重商品化を避ける。

## 7. ファイル、形式、プレビュー

### 7.1 制限

| 項目 | 現在値 |
|---|---:|
| 商品へ含めるファイル | 最大100件 |
| 追加時に検出するファイル | 最大300件 |
| 商品パッケージ総量 | 最大50 MiB |
| サンプルプレビュー | 最大6枚 |
| サムネイル長辺 | 最大640px |
| サンプル長辺 | 最大960px |

### 7.2 対応形式

- PiXiEEDrawプロジェクト: `pixiedraw-project`
- PNG: `png`
- PNGスプライトシート: `sprite-sheet-png`
- WebP: `webp`
- GIF: `gif`
- APNG: `apng`

ファイルとフォルダの入口は一つで、選択またはドラッグ&ドロップ後に形式を自動判定する。未対応形式も追加一覧には残すが、販売パッケージには含めない。検出済み対応形式は自動でONになり、出品者は不要形式だけをOFFにできる。

GIFはファイル選択・アップロード準備時に全フレームを確認し、全フレームで共通して完全一致する整数拡大だけを取り除く。例えば8倍ドットと4倍ドットのフレームが混在する場合は共通の4倍まで、1px単位の細部を持つフレームが1枚でもある場合は縮小しない。元の拡大フレーム群を販売用パッケージへ重複保持せず、検証済みの適正化データを以後のプレビュー、販売出力、PiXiEEDraw取り込みに使用する。

整数倍で拡大されたGIFは、出品画面へファイルを追加した時点で1px単位の適正解像度へ変換する。判定は合成後の全フレームに対して行い、すべてのフレームに共通して成立する最大倍率だけを使用する。

- 8倍と4倍のフレームが混在する場合は、共通して成立する4倍で縮小する。
- 共通倍率が1になる場合や、途中の1フレームでも整数倍構造が成立しない場合は変換しない。
- フレームごとに別倍率で縮小せず、配置、大きさ、動きを変えない。
- フレーム数、各フレーム時間、ループ回数を維持する。
- 再生成したGIFをもう一度デコードし、全フレームのRGBA、時間、ループ、寸法が一致した場合だけ元ファイルと置き換える。
- 置換後は、出品下書き、容量判定、プレビュー、SHA-256、マニフェスト、Storage送信のすべてで適正化後GIFを使う。
- マニフェストには元寸法、適正寸法、共通倍率、元サイズ、適正化後サイズ、元SHA-256を記録するが、元GIF本体はアップロードしない。

### 7.3 プレビュー

- プレビュー選択を「元作品から変更・追加した箇所」の入力より前に配置する。
- サムネイルを含むすべての購入前プレビューには、小さく薄い`PiXiEED SAMPLE`透かしを斜め格子状に高密度で入れる。
- 元ファイルと同じ永続URLは公開しない。
- 公開プレビューは非公開StorageからEdge Functionが短時間の署名URLを作る。
- 右クリック、ドラッグ、選択、長押し保存等をUIで抑止するが、ブラウザ表示データの完全な取得防止は保証しない。
- GIF・APNGの購入前サンプルは現状、静止画中心であり、透かし付き動画プレビューは未実装である。

## 8. タグ

- 最大5個。
- 1タグ最大24文字。
- Enter、カンマ、追加ボタンで確定する。
- 確定後はチップ型ボタンになり、クリックで削除できる。
- 大文字小文字を区別せず重複を除く。
- タイトル、説明、作者名、タグを横断検索する。
- タグは検索用メタデータであり、商品カード、商品詳細、購入済みカードには表示しない。

## 9. 価格とオプション

### 9.1 共通ルール

- 購入者の支払額は、選択したオプションを含めて最低500円。
- 販売者が入力する金額は販売価格1つだけであり、それが購入者の支払額になる。
- 金額は100円単位。
- 500円未満または100円単位でない入力は、画面とサーバーで上方向に補正する。
- 例: 450円は500円、550円は600円。
- 出品画面は500円から100円刻みの候補を提示し、上限までは直接入力もできる。
- 大元作品の画面上限は99,999,900円。Stripe購入上限は99,999,999円。

### 9.2 標準オプション

| オプション | 料金 | 状態 |
|---|---:|---|
| 商用・収益化利用 | 販売価格に含む | 有効 |
| グッズ・印刷販売 | 販売価格に含む | 有効 |
| クレジット表記不要 | 販売価格に含む | 有効 |
| ゲーム・アプリ利用 | - | 商用・収益化利用へ統合 |
| 動画・配信利用 | - | 商用・収益化利用へ統合 |

選択された標準オプションは価格を持たず、販売価格に含まれる。販売者は必要なものだけを選択できる。

### 9.3 派生許可

- 0円固定。
- 有料化できない。
- 許可した系列では正規購入1件につき派生出品権を原則1件作る。
- 派生作品も許可状態と系列条件を継承する。

### 9.4 限定販売

- ONにした場合だけ人数欄を表示する。
- 先着1〜100,000人。
- 限定販売オプション自体は最低100円、100円単位。
- 限定販売料金は購入価格と系列の必須価格へ加算する。
- Stripe Checkout開始時に商品行をロックし、支払済み数と有効な一時予約を合計して上限を確認する。
- 最後の1件へ同時操作が入っても販売数を超過させない。
- Stripe未接続の期限切れ予約は解放する。
- Stripe接続済み予約はWebhookの完了、失敗、期限切れで確定または解放する。
- 完売後も商品は削除せず、`SOLD OUT`として閲覧可能にする。
- 返金時は販売枠を戻す。支払い係争中は販売枠を占有したままにする。

### 9.5 カスタムオプション

- 最大10個。
- 名称1〜50文字。
- 説明最大200文字。
- 金額100〜10,000,000円、100円単位。
- 名称は大文字小文字を区別せず重複不可。
- 選択済みカスタムオプションは購入価格へ加算する。
- 系列条件としてスナップショットし、派生作品へ継承する。

## 10. 派生出品権と最低価格

### 10.1 権利

購入完了時、系列が派生販売を許可している場合だけ`market_derivative_rights`へ権利を作る。

- 購入記録1件につき原則1権利。
- `allowed_listing_count = 1`。
- 状態: `active`、`used`、`revoked`、`refunded`。
- 使用先商品IDを記録する。
- 同じ購入完了を再処理しても重複作成しない。

### 10.2 派生最低価格

派生商品の最低購入価格は次で決める。

```text
系列の大元価格合計 = 大元作品の販売価格 + 大元作品の必須オプション合計
派生最低購入価格 = 系列の大元価格合計 × 1.2
派生出品者が入力できる最低販売価格 = 派生最低購入価格 - 継承必須オプション合計
```

世代が増えるたびに1.2倍するのではなく、大元作品の基準から固定計算する。過去に確定した系列条件が100円単位でない場合、履歴保全のため既存値を保持する。

### 10.3 変更量

運営が主観的な「最低変更量」を設定する仕組みはない。ただし派生商品では、元作品から変更・追加した箇所の説明を必須とし、親商品、購入記録、未使用権利、系列最低価格をサーバーで確認する。

## 11. 公開と審査

### 11.1 即時公開

次をすべて満たす低リスクの大元作品は、パッケージ確定後に自己申告作品として即時公開する。

- 親商品を持たない大元作品。
- AI使用申告が「使用なし」。
- 必須の権利・規約・プライバシー確認がある。
- パッケージ、ハッシュ、形式、価格がサーバー検証を通る。

### 11.2 要審査

次は審査待ちにする。

- AI使用申告が「使用あり」の商品。
- すべての派生商品。
- 運営判断で確認が必要な商品。

承認にはファイル検査状態`clean`と元ファイルSHA-256が必要である。審査担当は`market_review_listing`で承認、却下、停止を行い、一般ユーザーはこのRPCを実行できない。

### 11.3 表示の区別

- PiXiEED形式／外部形式: ファイルの制作元・マニフェスト由来。
- AI使用あり／なし: 出品者の申告。
- 自己申告公開／**出品審査済み**: 商品公開までの確認経路。
- **販売者確認済み**: アカウントと受取設定の確認。

これらは別の情報であり、PiXiEED形式だから著作権が保証される、販売者確認済みだから全商品が審査済み、という意味ではない。

## 12. 商品一覧と詳細

### 12.1 カード

- サムネイル
- 公開用作者名
- お気に入り状態と集計件数
- 直接派生数
- 派生販売の許可状態
- 限定販売の残数または`SOLD OUT`
- 取り下げ済みの場合の`SOLD OUT`

作者名は非公開プロフィール行を直接公開せず、商品へ同期した公開用表示名を使う。お気に入り利用者IDは公開せず、各利用者は自分の状態だけをRLS経由で読み書きする。

### 12.2 検索・絞り込み・並び替え

- 検索: タイトル、説明、作者名、タグ。
- 絞り込み: 収録形式、価格、派生許可、大元作品のみ、派生作品のみ。
- 並び替え: 新着、価格、全商品の人気、派生商品の人気、直接派生数。

### 12.3 商品分類

実際に`pixiedraw-project`を収録する商品をPiXiEEDraw作品、それ以外の画像中心の商品を一般素材として表示する。分類、ファイル由来、AI申告、審査状態は別バッジで扱う。

## 13. 購入とStripe Checkout

### 13.1 Checkout開始

`market-create-checkout`が認証と商品状態を確認し、`market_create_purchase_intent_v1`で購入予約を作る。

- 金額はクライアント入力を信用せず、DBの商品・オプションからサーバー計算する。
- 最低100円、最大99,999,999円。
- 自分の商品は購入できない。
- 既に`paid`、`granted`、`disputed`の取得権がある商品は再購入できない。
- 非公開、停止、取り下げ、完売商品は購入できない。
- Checkout SessionはStripeの`payment`モード、JPY、商品1件で作る。
- 購入ID、購入者ID、商品IDをStripe metadataへ保存する。
- Session作成は冪等キーを使い、購入予約と`market_bind_stripe_checkout_v1`で結び付ける。
- Checkoutの有効期限は30分以上確保する。

### 13.2 Webhook

`market-stripe-webhook`だけがStripe署名を検証し、購入確定・取消・返金・係争処理を行う。イベントIDは`market_payment_events`で一度だけ処理する。

| Stripeイベント | 主な処理 |
|---|---|
| `checkout.session.completed` | PaymentIntent、Charge、実決済手数料を確認して購入確定 |
| `checkout.session.async_payment_succeeded` | 非同期支払いを購入確定 |
| `checkout.session.expired` | 未完了予約を取消 |
| `checkout.session.async_payment_failed` | 未完了予約を取消 |
| `account.updated` | Stripe Connect状態を同期 |
| `charge.refunded` | 全額返金時に購入権、台帳、派生権を反転 |
| `charge.dispute.created` | 係争状態へ変更し、分配を保留・反転 |
| `charge.dispute.closed` | 勝訴なら復元、敗訴なら反転を確定 |

ブラウザから`paid`購入、継承ロイヤリティー台帳、派生出品権を直接作成することはできない。

### 13.3 購入状態

- `pending`: Checkout待ち。
- `paid`: 支払い確定。
- `granted`: 管理者取得。
- `refunded`: 返金済み。
- `disputed`: 支払い係争中。
- `cancelled`: 期限切れまたは失敗。

## 14. 利用手数料と継承ロイヤリティー

### 14.1 PiXiEED利用手数料

| 購入価格 | 利用手数料率 |
|---:|---:|
| 500〜999円 | 10% |
| 1,000〜1,999円 | 9% |
| 2,000〜2,999円 | 8% |
| 3,000〜4,999円 | 7% |
| 5,000〜9,999円 | 6% |
| 10,000〜19,999円 | 5% |
| 20,000〜29,999円 | 4% |
| 30,000〜39,999円 | 3% |
| 40,000〜49,999円 | 2% |
| 50,000円以上 | 1% |

利用手数料は円未満を切り捨てる。決済実費はPiXiEED利用手数料とは別に扱う。

```text
作者分配対象額 = 購入価格 - Stripe実決済手数料 - PiXiEED利用手数料
```

### 14.2 系列への配分

- 現在の商品販売者: 深さ0。祖先分を引いた残額。
- 直接の親作者: 作者分配対象額の10%。
- 祖父母作者: 5%。
- 以後: 2.5%、1.25%と世代ごとに半減。
- 再帰上限: 60世代。
- 同じ受取人が複数世代にいる場合は内部台帳で合算する。

深い系列では祖先数により現在販売者の割合が変わるため、「必ず80%以上」とは保証しない。購入者の決済は1回だけで、作者ごとの配分はPiXiEED内部台帳で行う。

### 14.3 micro-yen台帳

`market_royalty_ledger`は`1円 = 1,000,000 micro-yen`で記録し、1円未満の継承ロイヤリティーも失わず繰り越す。

- `pending`: 購入確定後の保留中。
- `available`: 30日経過後に出金可能。
- `paid_out`: Stripe Transferへ反映済み。
- `reversed`: 返金・チャージバック等で取消。

送金済み取引が返金・敗訴となった場合はStripe Transfer Reversalを試行する。

## 15. 表示報酬

### 15.1 原資

表示報酬はPiXiEEDの売上の一部を原資として、運営が月ごとの表示報酬予算を設定し、その予算内で有効表示数に応じて配分する。

`PV × 固定単価`を無制限に支払う方式ではない。月次予算が0円または未設定なら、表示記録があってもその月の配分額は発生しない。

### 15.2 有効表示

- 商品詳細が前面・表示状態で10秒以上開かれた場合に候補化する。
- 記録する滞在時間は10〜3,600秒。
- 同じ閲覧者、同じ商品、同じ日付は1回だけ。
- ログイン中の作者本人による自己表示は除外する。
- 同一閲覧者は1日100商品まで。
- 匿名閲覧者は生の識別情報ではなくSHA-256化した閲覧者キーで重複排除する。

### 15.3 系列内配分

1件の有効表示に割り当てられた額は、表示作品と直系の祖先・子孫に含まれる世代数で均等に分ける。同じ世代に複数作品がある場合、その世代分を作品数でさらに均等に分ける。兄弟作品など、表示作品と祖先・子孫関係にない別枝へは配分しない。

例:

- `A → B`は2等分し、A世代とB世代へ配分。
- `A → B → C → D`なら4世代へ均等配分。
- C世代に`C・C2・C3`があれば、C世代の1枠を3等分。

世代と系列は記録済みの親子IDから確定し、画像類似度などから自動推測せず、別系列の作品を配分対象へ混ぜない。

対象月終了後、管理者が試算し、確定した配分だけを作者マイページへ表示する。確定行はスナップショットとなり、後から系列が増えても過去月を再計算しない。

### 15.4 作者マイページ

`market_my_reward_dashboard_v1`は6〜24か月の範囲で、次をサーバー集計して返す。

- 販売収益
- 継承ロイヤリティー
- 表示報酬
- 有効表示数
- 販売件数、継承ロイヤリティー対象販売件数
- 出金可能残高、保留残高

表示報酬は月次確定後に専用配分テーブルへ確定し、その未出金額を販売収益・継承ロイヤリティーの出金可能残高へ合算する。販売由来の`market_royalty_ledger`と表示報酬の配分行は別々に保持し、1回のStripe出金バッチではそれぞれの明細を監査可能な形で紐付ける。表示報酬は月次予算から確定した原資であり、購入の返金待ちには入れない。

## 16. PiXFiND等の有効プレイ記録

### 16.1 現在実装済み

PiXFiNDを最初の対象として、他のゲームにも拡張可能な`creator_reward_contents`と`creator_play_events`を持つ。

- コンテンツごとにアプリキー、コンテンツキー、作者、名称、有効状態、有効プレイ総数を保持する。

- 作者本人のログイン中の自己プレイは除外する。
- 同一閲覧者は1日100コンテンツまで。
- 匿名閲覧者キーはSHA-256で受け取る。
- 作者はマイページで各投稿の有効プレイ数を確認できる。
- 共通記録RPC: `creator_record_valid_play_v1`
- 作者統計RPC: `creator_my_play_stats_v1`

### 16.2 未接続

- 月次のプレイ報酬単価・配分計算。
- プレイ報酬を作者残高・Stripe出金へ接続する処理。

したがって現段階では「遊ばれた回数に応じて報酬額が確定する」状態ではなく、「将来、PiXiEEDの売上の一部を原資とするプレイ報酬へ接続できる有効プレイ記録がある」状態である。

## 17. 購入済み商品の受け取り

### 17.1 認可

`market-download`は毎回、本人の`paid`または`granted`取得権を確認する。

- マニフェストに含まれる販売対象ファイルだけを認可する。
- 利用者が選んだ形式を収録形式と照合する。
- 非公開Storageのファイルへ60秒の署名URLを発行する。
- ライブラリ用プレビューURLは300秒。
- 同じ利用者・同じ商品は1時間30回までの認可に制限する。
- `market_download_events`へ利用者、商品、追跡番号を監査記録する。

### 17.2 ZIP

ブラウザは認可済みファイルを取得し、SHA-256を照合してからZIPを生成する。ZIPには選択ファイル、商品マニフェスト、利用条件、追跡番号を含める。Storage URLや個人情報は含めない。

ZIP生成は無圧縮Store方式とCRC32を使う。ZIP32の単一ファイル・総サイズ上限を超える商品は生成できない。

### 17.3 GIF派生出力

購入済みGIFは元GIFに加え、端末内で次を生成できる。

- 全フレームPNG ZIP: `gif-frames-png`
- PNGスプライトシート: `gif-spritemap-png`

元GIFを認可取得後、ブラウザ内のGIFコーデックでフレーム展開する。整数倍拡大GIFは全フレーム共通倍率で1px単位へ補正してからPNGを生成し、`frames.json`またはSpriteMAPメタデータへ元寸法、出力寸法、倍率、フレーム数、時間、ループを記録する。フレーム数は削減しない。上限は1,000フレーム、スプライト総画素32M、Canvas一辺16,384px。生成PNGはSupabaseへ保存せず、購入者端末内だけで作る。APNGの同等フレーム展開は未実装である。

### 17.4 PiXiEEDrawで開く

取り込み優先順位は次の通り。

1. PiXiEEDrawプロジェクト
2. PNG
3. WebP
4. GIF
5. APNG
6. PNGスプライトシート

選んだ1ファイルを同一オリジンのIndexedDB `pixieed-market-import-v1`へ最大5分だけ置き、PiXiEEDrawの新規プロジェクトとして開く。URLへファイル本体を埋め込まない。

PiXiEEDrawの新しい空キャンバスはINDEXカラーで開始する。保存済みプロジェクトは保存された`colorMode`を優先するため、RGBで保存した作品をINDEXへ自動変換しない。画像・GIF取り込みも取り込み元のRGB表現を維持する。

旧バージョンで保存済みの大きなプロジェクトは、読込時に全フレーム・全レイヤーの境界を確認する。総画素走査対象が1,000,000画素以上で、完全一致する共通整数倍率があり、選択・ミラー・シミュレーション・共有編集・編集済みタイムラプスを壊さない場合だけ、パレット同期前にネイティブドット寸法へ適正化して同じ端末内プロジェクトへ自動保存する。条件を満たさない場合は元寸法を維持する。

GIFを開く場合、PiXiEEDrawも全フレームに共通して成立する最大整数倍率を検査し、適正解像度のRGBAだけをプロジェクトへ保持する。元GIFバイナリと元解像度の全フレームは`.pixieedraw`や自動保存へ保持しない。204フレーム等のフレーム数と各時間は維持する。

## 18. 出品取り下げ

出品者本人は`market_withdraw_my_listing_v1`で公開商品を取り下げられる。

- 取り下げ後も商品行と公開詳細を削除しない。
- 一覧と詳細では`SOLD OUT`扱いにする。
- 新しい購入とCheckoutを停止する。
- 購入済み・管理者取得済み利用者の再ダウンロード権を維持する。
- 派生許可、既存の派生出品権、系列、親子関係を維持する。
- 継承ロイヤリティー、表示報酬、販売履歴を維持する。
- `withdrawn_at`と理由を記録する。

取り下げは削除ではなく、系列と購入者保護を維持した販売停止である。

## 19. 管理者取得

`market_grant_admin_asset_access_v1`は、運営確認用に`granted`取得権を作る。

- Stripe決済なし。
- 販売数、限定枠、販売収益、PiXiEED利用手数料、継承ロイヤリティーへ加算しない。
- 派生販売可能商品では、通常購入と同じ派生出品権を作る。
- 既存の支払済み購入や進行中Checkoutを上書きしない。
- 管理者権限と監査記録を必須にする。

## 20. Stripe Connectと出金

### 20.1 受取設定

`market-stripe-connect`は日本の個人向けExpressアカウントを作成し、Transfers capabilityを使う。`account.updated`で次を同期する。

- 詳細提出状態
- 入金可否
- 未完了要件
- 無効理由
- アカウント状態

受取設定が未完了でも出品・販売・残高蓄積はできるが、出金はできない。

### 20.2 出金条件

- 販売収益・継承ロイヤリティーの30日保留を終えた額と、月次確定済み表示報酬の合計が5,000円以上。
- Stripe Connectの受取設定が完了。
- Asia/Tokyo基準で月1回まで。

申請前に、利用可能額、Stripe Transfer費用、銀行口座への入金費用、当月の受取口座利用料、最終Transfer額を表示する。現在の見積式は次の通り。

- Stripe Transfer費用: 0.25%を切り上げ。
- 銀行口座への入金費用: 0.25%を切り上げ + 250円。
- 当月の受取口座利用料: 200円。

`market-request-payout`は申請、販売・継承台帳と表示報酬明細の確保、Stripe Transfer、確定を冪等に実行する。失敗時は申請を失敗状態にし、台帳の不整合を避ける。実際のStripe料金改定時は、実装と本書の両方を更新する必要がある。

## 21. Storage、RLS、セキュリティ

### 21.1 Storage

- 非公開bucket: `market-private`。
- 出品者は自分のユーザーIDで始まる許可パスだけをアップロード・読取・削除できる。
- 審査担当は審査対象を読める。
- 公開閲覧者へ元ファイルのStorageパスを返さない。
- 公開プレビューと購入ファイルはEdge Functionの短時間署名URLだけを使う。

### 21.2 DB境界

- 価格、系列、購入状態、継承ロイヤリティー、派生権をブラウザから直接書き換えさせない。
- `SECURITY DEFINER` RPC内で`auth.uid()`、状態、価格、権利、回数を再確認する。
- 旧バージョンRPCの一般実行権を剥奪する。
- 公開一覧はテーブルの広いSELECTではなく、公開用RPCが必要列だけを返す。
- 購入・お気に入り・出品一覧は本人行だけをRLSまたは本人限定RPCで返す。
- Service Role専用の決済完了RPCをブラウザへ公開しない。

### 21.3 不正・再配布対策の限界

非公開Storage、短時間URL、購入権検証、ハッシュ検証、取得回数制限、サムネイルを含む高密度透かし、監査追跡、利用規約を重ねる。ただし正規購入後の画面撮影や外部再配布を技術だけで完全には防げない。違反時は監査記録、停止、権利取消、返金・係争処理、法的対応を組み合わせる。

- タグが最大5個で止まり、チップクリックで削除できる。

### 購入

- Stripe表示額がDB再計算した購入価格と一致する。
- 同時購入でも限定枠を超えない。
- Webhook再送で購入権や継承ロイヤリティーが重複しない。
- 返金・係争で購入権、派生権、台帳、限定枠が正しく反転する。

### 配送

- `paid`または`granted`だけが収録ファイルを取得できる。
- 選択形式以外のStorageパスを要求できない。
- 取得後のSHA-256がマニフェストと一致する。
- GIFの全フレームPNG、スプライトシートが上限内で生成できる。
- 整数倍GIFがファイル追加時に適正化され、全フレーム共通倍率、画素完全一致、時間、ループを確認してからStorageへ送信される。
- 途中フレームだけ倍率が違うGIFで、共通倍率を超えた縮小やフレームごとの個別縮小が行われない。

### 系列・取り下げ

- 派生出品が購入済み商品の未使用権利からしか開始できない。
- 派生最低価格と継承オプションがサーバーで固定される。
- 取り下げ後は`SOLD OUT`となり、新規購入だけが止まる。
- 既存購入者の取得、派生権、系列、継承ロイヤリティー、表示報酬履歴が維持される。

### 報酬・出金

- 表示報酬の原資表記が「PiXiEEDの売上の一部」で統一されている。
- 自己表示、自己プレイ、同日重複、1日100件超が除外される。
- 表示報酬は月次確定後に作者画面へ出て、未出金分は販売・継承ロイヤリティーと同じ出金可能残高へ合算される。
- プレイ数は記録されるが、未実装の報酬額を表示しない。
- 30日保留、5,000円最低額、月1回、Connect準備完了を満たす場合だけ出金できる。

## 27. 仕様判断の優先順位

本番動作を判断するときは、次の順に確認する。

1. 最新Supabase migration、Edge Function、実際に呼ばれるフロントエンドコード。

構想資料は目標と背景を残すための文書であり、すべてが実装済みであることを意味しない。本書もコード変更後は自動更新されないため、価格、Stripe料金、RPCバージョン、審査条件を変更した場合は同じコミットで更新する。
